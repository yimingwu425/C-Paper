# RELEASE_CANDIDATE_AUDIT

## Purpose

This file is the current release-candidate audit matrix for the active native macOS product line. It maps the long-running quality goal onto concrete evidence so future contributors can tell which claims are already proven, which are only partially proven, and which gaps still remain.

## Current Audit Baseline

- Audit baseline version: `6.0.7`
- Latest documented full proof: `2026-06-18` release-candidate audit for the previous native release line.
- Current status: the evidence matrix remains the active release-candidate checklist, but a fresh `6.0.7` RC pass must be run before tagging a new release.
- Required release-candidate proof:
  - `bash scripts/run_native_release_audit.sh --release-candidate`
  - If live provider behavior needs confirmation outside the package audit path: `RUN_LIVE_SOURCE_TESTS=1 swift test --jobs 1 --filter LiveSourceTests`

## Gaps That Still Matter

### P1 — No automated UI-level end-to-end workflow assertions

Current proof for search, preview, download, and update state transitions is strong at the service/model/script level. This now includes pure presentation coverage for route-scoped search/batch notices, downloads-page controls, root-level update/support notices and alerts, rendered interaction assertions for sidebar route switching, root toolbar download routing, visible search execution and retry-after-failure recovery, keyboard submit routing for the search and batch filter surfaces, the searchable `SubjectPicker` popover’s selection and dismiss/reopen reset flow, root-surface search-to-download and batch-preview-to-download chains, search-page and batch-page download-notice retry recovery after startup-stage failures, batch-page preview retry recovery and download initiation into the downloads route, preview failure retry and re-download recovery actions, downloads-center cancellation/retry/integrity-repair actions, download-recovery notice escalation into the settings sheet, root toolbar settings-sheet presentation, root error-alert dismissal behavior, startup-failure diagnostic copy plus contextual support-folder failure alert handling, startup-failure retry dispatch plus coordinator recovery semantics, root startup update prompt primary actions, the root update-notice retry-download path, settings-page update/support actions including live update download initiation, real clipboard copying from visible diagnostics, and AppKit menu-driven refresh/recovery routing through installed menu items. Menu proof now also asserts that root workflow commands stay blocked while the settings sheet or startup update prompt is modal, that error-alert state still keeps diagnostic-copy access while workflow-changing commands are gated, that the installed “复制最近诊断” menu item copies the redacted latest report and still works while a root error alert is visible, that installed “显示支持目录” can still escape the same root error alert into a visible contextual support notice instead of failing silently, that installed “检查更新...” can still surface a visible retryable root notice after a failed manual check, and that installed “显示下载文件夹” can still create a missing valid directory without surfacing a false failure state. Even so, the repo still lacks broader UI interaction coverage for the full human-visible workflow wiring. That means focus behavior, first-responder semantics, and richer multi-step AppKit/SwiftUI chains can still regress in ways that are subtler than the current suite can prove.

### P1 — Full release-candidate audit still depends on opt-in external checks

The stronger RC pass is now available through one explicit command, but it still depends on optional external or slower checks rather than the default command path:

- `bash scripts/run_native_release_audit.sh --release-candidate`

Those checks remain intentionally outside the default path because they are slower or externally unstable, so future release candidates still need an explicit decision to rerun them. The main remaining gap is no longer “remember the right flag combination”, but that the stronger RC gate is still not the default fast audit path.

### P2 — Credentialed signing/notarization remains conditionally verified

The repo documents the contract and scripts for Developer ID signing and notarization, but that branch cannot be proven in ordinary local development without Apple credentials.
