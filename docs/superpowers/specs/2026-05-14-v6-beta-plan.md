## 一、代码库现状分析

**技术栈**: Python 桌面应用，pywebview 框架（Python 后端 + HTML/JS/CSS 前端）

**后端结构** (`src/backend/`):
- `api.py` — 主 API 类，Python-JS 桥接层，所有前端可调用方法均在此类上
- `engine.py` — DownloadEngine，含 TokenBucket 限速和 CircuitBreaker 熔断
- `parser.py` — 网络请求、文件名解析、试卷分组
- `cache.py` — JSON 文件缓存和持久化（`~/.cie_cache/`）
- `const.py` — 常量定义（BASE_URL, CACHE_DIR, VERSION 等）
- `limiter.py` — TokenBucket 和 CircuitBreaker 实现
- `plugin_manager.py` — 插件系统（Python 钩子 + 外部命令）
- `updater.py` — 自动更新检查

**前端结构**: `ui_v2.html` + `ui_v2.js` + `ui_v2.css`，左侧 rail 导航栏（搜索/批量下载/下载列表三个面板），右侧面板切换。

**测试**: pytest，测试文件在 `tests/` 目录，使用 `tmp_path` fixture 和 `monkeypatch`。

**关键模式**: API 类所有方法返回 `json.dumps(...)` 字符串，前端通过 `pywebview.api.xxx()` 调用并 `JSON.parse()` 解析。异步操作通过 `threading.Thread` + 轮询 `get_status()` 实现。

### 2.2 ocr_engine.py — OCR 引擎

**职责**: 从 PDF 提取文本，支持全文搜索和 AI 分析的文本输入。

**核心类与函数签名**:

```python
class OCREngine:
    """PDF OCR 引擎 — PDF → 图片 → 预处理 → Tesseract → 后处理 → 结构化文本"""

    def __init__(self, tesseract_cmd: str = "", lang: str = "eng",
                 dpi: int = 300, cache_dir: str = ""):
        """
        Args:
            tesseract_cmd: Tesseract 可执行文件路径（空则自动检测）
            lang: OCR 语言（eng, chi_sim 等）
            dpi: PDF 转图片的 DPI
            cache_dir: OCR 结果缓存目录
        """

    def extract_text(self, pdf_path: str, progress_cb: Callable = None) -> OCRResult:
        """
        从 PDF 提取全文文本。
        Args:
            pdf_path: PDF 文件路径
            progress_cb: 进度回调 (current_page, total_pages, message)
        Returns:
            OCRResult(full_text, pages, questions, metadata)
        """

    def extract_questions(self, pdf_path: str, progress_cb: Callable = None) -> list[Question]:
        """
        提取并按题号分割题目。
        Returns:
            list[Question(number, text, marks, page, formula_positions)]
        """

    def _pdf_to_images(self, pdf_path: str) -> list[Image.Image]:
        """PDF 每页转为高 DPI 图片（使用 pdfplumber）"""

    def _preprocess_image(self, img: Image.Image) -> Image.Image:
        """图片预处理：灰度化 → 去噪 → 二值化 → 倾斜校正"""

    def _ocr_image(self, img: Image.Image) -> str:
        """单张图片 Tesseract OCR 识别"""

    def _postprocess_text(self, raw_text: str) -> str:
        """后处理：数学符号修正、排版整理、公式标记为 [formula]"""

    def _split_questions(self, text: str) -> list[Question]:
        """按题号正则分割文本为独立题目"""


@dataclass
class OCRResult:
    full_text: str
    pages: list[str]           # 每页文本
    questions: list[Question]   # 按题号分割
    metadata: dict              # page_count, dpi, elapsed_ms 等

@dataclass
class Question:
    number: str                 # 题号，如 "1", "2(a)"
    text: str                   # 题目文本
    marks: str                  # 分值
    page: int                   # 所在页码
    formula_positions: list     # 公式占位符位置
```

**输入**: PDF 文件路径
**输出**: OCRResult（含全文、分页、分题结果）

### 任务 2: OCR 引擎实现

**文件**: `src/backend/ocr_engine.py`（新增）

**核心实现要点**:
- 使用 `pdfplumber` 将 PDF 每页导出为 PNG 图片（300 DPI）
- 使用 `Pillow` 进行图片预处理（灰度化、自适应二值化、去噪）
- 使用 `pytesseract` 调用 Tesseract 5.x 进行 OCR
- 正则匹配题号模式（如 `1`, `2(a)`, `Question 3`）进行分题
- 数学公式区域标记为 `[formula]` 占位符
- OCR 结果通过 `cache.py` 的 `write_json` 机制缓存到 `~/.cie_cache/ocr_results/`

**输入**: PDF 文件路径
**输出**: OCRResult 数据类

### 任务 3: LLM 调用封装

**文件**: `src/backend/ai_analyzer.py`（新增）

**核心实现要点**:
- `OpenAIProvider` 使用 `openai` 库（兼容 OpenAI 和通义千问 API）
- `AnthropicProvider` 使用 `anthropic` 库或直接 HTTP 调用
- API Key 存储在 `~/.cie_cache/ai_config.json`，不上传服务端
- 分析 prompt 包含: system prompt（角色设定 + 输出 JSON schema）+ user prompt（试卷文本）
- 使用 `json.loads` + schema 校验解析 LLM 返回
- 分析结果缓存到 `~/.cie_cache/ai_results/`，key 为试卷文件名的 hash
- 支持 streaming 输出（通过 `progress_cb` 回调）

**输入**: OCR 文本 + 用户配置的 API Key/模型
**输出**: AnalysisResult 数据类

### 任务 4: 智能去重引擎

**文件**: `src/backend/dedup_engine.py`（新增）

**核心实现要点**:
- 使用 `sentence-transformers` 加载 `all-MiniLM-L6-v2` 模型（80MB）
- 模型首次下载缓存到 `~/.cie_cache/models/`
- 使用 `faiss-cpu` 建立向量索引（IndexFlatIP，内积即余弦相似度，因向量已归一化）
- 索引持久化到 `~/.cie_cache/dedup_index/`（faiss.write_index + pickle 元数据）
- 新试卷加入时增量更新索引，无需全量重建
- 相似度阈值: >0.80 标记"高度相似"，>0.65 标记"可能相关"

**输入**: 题目文本列表
**输出**: SimilarMatch 列表

### 任务 5: 全文搜索引擎

**文件**: `src/backend/fts_engine.py`（新增）

**核心实现要点**:
- SQLite 数据库路径: `~/.cie_cache/cpaper_fts.db`
- FTS5 虚拟表: `papers_fts(paper_id, question_number, text, subject, year)`
- 辅助表: `papers_meta(paper_id, subject, year, season, filename, indexed_at)`
- 使用 `porter` tokenizer 支持英文词干提取
- 搜索结果通过 FTS5 的 `snippet()` 函数返回匹配片段

**输入**: 搜索关键词
**输出**: SearchResult 列表

## 四、LLM 调用封装详细设计

### 4.1 多 Provider 架构

```
AIAnalyzer
  ├── configure(provider, api_key, model, base_url)
  │     ├── "openai"  → OpenAIProvider(api_key, base_url="https://api.openai.com/v1")
  │     ├── "anthropic" → AnthropicProvider(api_key)
  │     └── "qwen"    → OpenAIProvider(api_key, base_url="https://dashscope.aliyuncs.com/compatible-mode/v1")
  │
  ├── analyze_paper(text, paper_info)
  │     ├── _build_analysis_prompt(text, paper_info)
  │     ├── provider.chat(messages, model)
  │     └── _parse_llm_response(raw)
  │
  └── test_connection()
        └── provider.chat([{"role":"user","content":"Say OK"}], model)
```

### 4.2 API Key 安全存储

API Key 存储在 `~/.cie_cache/ai_config.json`，结构:
```json
{
  "provider": "openai",
  "api_key": "sk-xxx...",
  "model": "gpt-4o-mini",
  "base_url": ""
}
```

前端 `get_ai_config()` 返回时将 API Key 脱敏为 `sk-***xxx`（仅显示前3后3字符）。

### 4.3 Prompt 设计

**System Prompt**:
```
你是一个 CIE (Cambridge International Education) 试卷分析专家。
请分析以下试卷文本，返回严格的 JSON 格式结果。

输出 JSON schema:
{
  "paper_info": {"subject": str, "year": int, "season": str, "paper_number": int, "total_marks": int, "question_count": int},
  "topics": [{"name": str, "questions": [int], "total_marks": int}],
  "difficulty_distribution": {"easy": int, "medium": int, "hard": int},
  "repeated_from_previous": [{"question": int, "similar_to": str, "similarity": float}],
  "summary": str
}

注意:
1. summary 用中文撰写
2. topics 按考点分类（如 Differentiation, Integration, Statistics 等）
3. difficulty 基于题目复杂度和知识点综合判断
4. 如果无法确定某些字段，使用 null
5. 仅返回 JSON，不要包含其他文本
```

### 4.4 错误处理

| 错误类型 | 处理方式 |
|---------|---------|
| API Key 无效 (401) | 返回 `{"ok": false, "error": "API Key 无效，请检查配置", "error_type": "auth"}` |
| 模型不存在 (404) | 返回明确错误信息，建议用户检查模型名称 |
| 请求过频 (429) | 指数退避重试（最多 3 次） |
| 网络超时 | 30 秒超时，提示检查网络/代理 |
| 返回非 JSON | 尝试从文本中提取 JSON 块（regex `\{[\s\S]*\}`），失败则报错 |
| Token 超限 | 截断试卷文本（保留前 12000 tokens），提示用户 |

## 五、OCR 工作流详细设计

### 5.1 处理流程

```
PDF 文件路径
  │
  ▼
[1] pdfplumber.open(pdf_path)
  │  逐页 page.to_image(resolution=300)
  │  导出为 PIL.Image
  ▼
[2] 图片预处理 (Pillow + numpy)
  │  ├── 灰度化: image.convert("L")
  │  ├── 去噪: ImageFilter.MedianFilter(size=3)
  │  ├── 二值化: 自适应阈值 (Otsu's method)
  │  └── 倾斜校正: 可选 (pytesseract image_to_osd)
  ▼
[3] pytesseract.image_to_string(image, lang="eng",
  │    config="--oem 3 --psm 6")
  │  OEM 3 = 默认引擎（LSTM + 传统）
  │  PSM 6 = 假设为统一文本块
  ▼
[4] 后处理
  │  ├── 数学公式检测: 正则匹配常见公式模式
  │  │   (积分号 ∫, 分式, 根号 √, 上下标等)
  │  │   → 替换为 [formula] 占位符
  │  ├── 特殊字符修正: OCR 常见错误映射
  │  │   (如 "l" ↔ "1", "O" ↔ "0" 在数学上下文中)
  │  └── 排版整理: 去除多余空行, 合并断行
  ▼
[5] 分题处理
  │  正则: r'(?:Question\s+)?(\d+[\(]?[a-z][\)]?)\s*[\[\.]'
  │  按题号分割为 Question 对象列表
  ▼
[6] 缓存结果
     write_json(~/.cie_cache/ocr_results/{hash}.json, result)
```

### 5.2 性能优化

| 优化点 | 方案 |
|-------|------|
| 大 PDF 处理 | 分页处理，每页完成后回调进度 |
| 重复处理 | 缓存机制：PDF 文件 hash → OCR 结果 |
| 内存占用 | 处理完一页立即释放图片对象 |
| 并发 | 单 PDF 内串行（Tesseract 不支持并发），多 PDF 可并行 |

### 5.3 数学公式处理策略

CIE 数学试卷公式密集。Tesseract 对纯文本准确率 >95%，但对数学公式识别率有限。

策略:
1. **检测**: 通过字符特征（特殊 Unicode、上下标、分数结构）检测公式区域
2. **标记**: 将公式区域替换为 `[formula]` 占位符
3. **保留**: 在 Question 对象中记录公式在原文中的位置
4. **展示**: UI 中公式位置显示为灰色占位块，鼠标悬停提示"公式区域"

## 六、去重工作流详细设计

### 6.1 处理流程

```
OCR 提取的题目列表 (list[Question])
  │
  ▼
[1] 文本预处理
  │  ├── 去除 [formula] 占位符（保留上下文）
  │  ├── 统一小写
  │  └── 去除多余空白
  ▼
[2] 生成 Embedding
  │  model = SentenceTransformer("all-MiniLM-L6-v2")
  │  embeddings = model.encode(texts, normalize_embeddings=True)
  │  # 输出: numpy array, shape (n, 384), dtype float32
  ▼
[3] FAISS 索引操作
  │  ├── 加载已有索引 (faiss.read_index)
  │  ├── 新增向量 (index.add(embeddings))
  │  └── 保存索引 (faiss.write_index)
  ▼
[4] 相似度检索
  │  D, I = index.search(query_embedding, top_k)
  │  # D: 相似度分数, I: 索引位置
  │  # 过滤 threshold < 0.65 的结果
  ▼
[5] 结果映射
     索引位置 → 元数据 (paper_id, question_number, text)
     → 构建 SimilarMatch 对象列表
```

### 6.2 索引结构

```
~/.cie_cache/dedup_index/
  ├── questions.index      # FAISS IndexFlatIP (384 维)
  ├── questions.meta.pkl   # list[dict] 元数据 (paper_id, number, text, subject, year)
  └── papers.set.pkl       # set[str] 已索引的 paper_id 集合
```

### 6.3 增量更新 vs 全量重建

- **增量更新** (`add_paper`): 新试卷加入时，生成 embedding → `index.add()` → 保存。O(n) 复杂度。
- **全量重建** (`build_full_index`): 从 SQLite 中所有已 OCR 的试卷重建。仅在索引损坏或用户手动触发时使用。

### 6.4 性能估算

| 操作 | 数据量 | 预计耗时 | 说明 |
|------|--------|---------|------|
| 模型加载 | 80MB | 2-5 秒 | 首次从磁盘加载 |
| embedding 生成 | 10 题 | <1 秒 | CPU 推理 |
| 索引检索 (1000 题) | 384 维 | <10ms | FAISS 扁平索引 |
| 索引检索 (10000 题) | 384 维 | <50ms | FAISS 扁平索引 |
| 全量重建 (1000 题) | - | 3-10 分钟 | 含所有 OCR 试卷 |

### 8.4 性能优化策略

| 优化点 | 方案 |
|-------|------|
| OCR 结果缓存 | PDF 文件 MD5 hash → OCR 结果 JSON，避免重复 OCR |

| Embedding 缓存 | 已生成的 embedding 存入 FAISS 索引，增量更新 |
| 延迟加载 | sentence-transformers 模型在首次使用时才加载，不在启动时 |
| 后台线程 | OCR、AI 分析、索引构建均在后台线程执行，不阻塞 UI |
| 内存管理 | 大 PDF 处理完一页立即释放图片；FAISS 索引使用内存映射 |
| FTS5 优化 | 使用 `content=` 选项减少存储空间；定期 `optimize` |

## 十一、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Tesseract 数学公式识别差 | OCR 质量下降 | 公式标记为 `[formula]`，保留截图，后续迭代 |
| sentence-transformers 模型下载失败 | 去重功能不可用 | 提供离线模型包下载链接，支持手动指定模型路径 |
| LLM API 返回格式不稳定 | 分析结果解析失败 | 多层容错：直接 JSON 解析 → regex 提取 → 降级为纯文本展示 |
| faiss-cpu 在某些平台编译困难 | 安装失败 | requirements.txt 中标注 faiss-cpu 版本范围，README 提供备选方案（如 Annoy） |
| 大量试卷首次索引耗时长 | 用户体验差 | 后台线程 + 进度条，支持中断和续传 |
